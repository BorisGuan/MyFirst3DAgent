# 机甲模型细节设计 Agent 产品设计计划书

## 1. 产品定位

本项目当前已经完成了基础意图理解层：可以读取抽象机甲模型上下文，识别用户命令类型，并输出结构化建模修改计划。下一阶段的产品目标不是做通用文生 3D 或图生 3D 工具，而是做一个面向机甲、拼装模型和 3D 打印模型创作者的垂直设计 Agent。

产品核心定位：

> 为已有机甲草模、部件库或设计概念提供细节丰富、分件规划、风格统一和可制作性检查的 AI 设计助手。

它更像是模型设计师的副驾驶，而不是自动替代建模软件。通用 3D AIGC 工具负责生成初稿，本工具负责让初稿变得更像一个合理、可制作、风格一致的机甲模型。

## 2. 目标用户

### 2.1 核心用户

- 机甲模型设计爱好者
- 3D 打印手办/机甲模型创作者
- Blender、ZBrush、Fusion 360 等工具的中初级用户
- 想把 AI 生成草模继续加工成可展示或可打印模型的创作者

### 2.2 次级用户

- 独立游戏开发者
- 科幻道具/机械设定设计师
- 3D 资产外包团队
- 教学场景中的 3D 建模学习者

## 3. 用户痛点

### 3.1 通用 3D 生成工具的不足

- 生成结果通常是整体 mesh，不知道如何合理分件。
- 对机甲装甲、关节、喷口、武器挂点等结构语义理解不足。
- 细节风格容易不统一，例如头部、胸甲、背包像不同作品里的东西。
- 不会主动考虑打印厚度、活动间隙、支撑、装配公差。
- 很难通过自然语言精确描述“某个部位加什么类型的机械细节”。

### 3.2 传统建模流程的门槛

- 新手不知道哪些位置适合加刻线、散热口、喷口、液压杆。
- 不知道高密度细节会不会影响 1/144、1/100 等比例下的制作效果。
- 缺少从“想法”到“可执行建模步骤”的中间层。
- 很难持续维护一套机甲设计语言和部件命名体系。

## 4. 产品目标

### 4.1 近期目标

先交付一个用户能马上使用的全链路 MVP：用户输入自然语言机甲细节设计需求，系统输出机器可读的机甲设计操作蓝图，并同步给出设计师可读的执行摘要和风险提示。在这个闭环内，把当前简单的 `target_part + action + detail_type + density` 计划升级成更稳定的中间指令格式，支持风格、对称、比例、分件、可制作性约束和执行步骤。

### 4.2 中期目标

构建一个可持续扩展的机甲模型设计 Agent 框架，能够：

- 理解更细的部件层级和部件关系。
- 生成可执行或半可执行的建模操作计划。
- 对计划进行设计审查和可制作性检查。
- 为 Blender 等建模工具输出操作脚本、标记或辅助数据。

### 4.3 长期目标

形成一个面向机甲模型制作的 AI 工作流：

1. 输入草模、部件清单或设计想法。
2. Agent 生成机甲设计操作蓝图。
3. Agent 进行分件和制作风险检查。
4. Agent 输出 Blender/Fusion 可执行步骤或脚本。
5. 用户在建模软件中确认、调整并迭代。

## 5. 产品边界

### 5.1 当前不做

- 不做底层通用文生 3D 大模型。
- 不直接承诺生成完整可动拼装模型。
- 不直接复刻受版权保护的具体商业机体。
- 不在第一阶段处理真实 mesh 的复杂布尔、重拓扑和 UV。

### 5.2 当前重点做

- 抽象机甲语义理解。
- 部件级操作蓝图规划。
- 设计风格和比例约束表达。
- 分件建议和可制作性风险提示。
- 后续建模工具集成的结构化接口。

## 6. 核心功能模块

### 6.1 机甲设计操作蓝图

当前 `OperationPlan` 需要扩展为更完整的机器可读操作蓝图。它不是普通说明文档，而是自然语言和后续建模执行层之间的中间指令格式。

建议字段：

- `target_part`: 目标部件。
- `operation`: 具体操作，例如添加刻线、添加喷口、添加散热口、增加装甲分层。
- `detail_type`: 细节类型。
- `style`: 设计风格，例如锐利、厚重、军武、写实、动画风、高机动。
- `density`: 细节密度。
- `symmetry`: 对称策略，例如仅目标部件、左右镜像、中心对称、成组同步。
- `scale`: 制作比例，例如 1/144、1/100、1/60。
- `placement_zones`: 建议放置区域。
- `constraints`: 打印、装配、比例、活动间隙等约束。
- `steps`: 可执行设计步骤。
- `risk_notes`: 风险提示。

### 6.2 设计分类扩展

当前细节类型只有三类，需要扩展为机甲模型专用 taxonomy。

建议新增类型：

- 装甲刻线
- 装甲分层
- 分件线
- 散热口
- 喷口/推进器
- 液压杆
- 管线
- 铆钉/螺栓
- 传感器
- 武器挂点
- 内构外露
- 关节保护件
- 战损缺口
- 表面旧化
- 卡榫/连接点

### 6.3 部件关系图

需要让 Agent 理解部件不是平铺列表，而是有层级和关联。

示例关系：

- `body` 包含 `chest_armor`、`cockpit_hatch`。
- `backpack` 包含 `booster`、`thruster`、`fuel_tank`、`wing_unit`。
- `arm` 包含 `left_arm`、`right_arm`、`hand`、`shoulder_armor`。
- `leg` 包含 `left_leg`、`right_leg`、`knee_armor`、`foot`。

该模块用于支持成组编辑、对称编辑、上下文联动和风险检查。

### 6.4 设计风格记忆

Agent 需要维护项目级设计风格，避免每次操作风格漂移。

可维护字段：

- 主风格：军武、写实、动画、厚重、高机动、竞速、复古。
- 细节密度：低、中、高。
- 装甲语言：直线、斜切、曲面、层叠、外露内构。
- 制作目标：展示模型、3D 打印、游戏资产、概念验证。
- 比例目标：1/144、1/100、1/60 或非比例。

### 6.5 设计审查

设计审查模块用于检查计划是否合理。

审查维度：

- 目标部件是否适合该细节类型。
- 细节密度是否过高。
- 是否破坏左右对称或成组一致性。
- 是否可能影响关节活动。
- 是否可能导致打印厚度不足。
- 是否与项目设计风格冲突。
- 是否需要拆分成多个步骤执行。

### 6.6 可制作性检查

面向 3D 打印和模型制作场景，输出风险提示而不是直接替用户做 CAD 决策。

检查项：

- 最小刻线宽度。
- 最小凸起厚度。
- 天线、喷口、管线等细件是否过细。
- 活动关节是否需要预留间隙。
- 装甲分件是否有合理切割边界。
- 是否需要支撑或拆件打印。

### 6.7 建模工具集成接口

第一阶段不直接做复杂几何，但需要为后续 Blender 集成预留接口。

可输出形式：

- JSON 操作计划。
- 人类可读步骤清单。
- Blender Python 操作草案。
- 标记点、区域、注释对象。
- Geometry Nodes 参数草案。

## 7. 版本路线图

### V0.1 已完成：基础意图理解层

已有能力：

- 命令分类。
- 抽象模型上下文加载。
- 部件别名识别。
- 简单意图解析。
- 简单操作计划校验。
- 上下文查询和交互记录。

### V0.2：可用全链路 MVP

目标：打通从用户自然语言输入到机甲设计操作蓝图输出的完整闭环，让用户可以马上用设计师摘要辅助手工建模，也让后续 Blender/Fusion 执行层可以复用同一份结构化蓝图。

开发内容：

- 扩展 `OperationPlan` schema。
- 扩展 `design_taxonomy.json`。
- 更新 intent prompt。
- 更新 planner 校验逻辑。
- 增加 mock parser 对新字段的默认推断。
- 增加示例输入输出。
- 增加设计师可读摘要输出。
- 增加基础风险提示。
- 增加命令行演示链路。

验收标准：

- 用户输入“给胸甲加一些锐利的分件线，适合 1/144 打印”时，系统能输出包含目标部件、操作、风格、比例、放置建议、执行步骤和风险提示的机器可读操作蓝图，并提供设计师可读摘要。

### V0.3：部件关系和成组编辑

目标：让 Agent 理解部件层级和对称关系。

开发内容：

- 增加部件关系配置。
- 支持左右镜像和成组同步。
- 支持父部件到子部件的目标展开。
- 支持“所有推进器”“两条腿”“左右肩甲”等集合表达。

验收标准：

- 用户输入“给两条腿加同样的机械细节”时，系统能识别为 `left_leg` 和 `right_leg` 的镜像/成组计划。

### V0.4：设计审查和可制作性检查

目标：从“生成计划”升级为“生成并审查计划”。

开发内容：

- 增加 plan reviewer。
- 增加制作约束配置。
- 输出风险等级和修正建议。
- 支持用户询问“这个设计有什么问题”。

验收标准：

- 用户要求在 V 字天线上增加大量刻线时，系统能提示细节过密、结构过细、打印风险高。

### V0.5：Blender 辅助输出

目标：开始连接建模软件，但先做低风险辅助功能。

开发内容：

- 输出 Blender Python 草案。
- 在场景中创建部件标记或注释。
- 为目标部件生成操作 checklist。
- 支持导出 plan JSON 文件。

验收标准：

- 用户选择一个部件后，Agent 能生成对应的 Blender 注释对象或操作清单，帮助用户手工建模。

### V1.0：机甲模型设计副驾驶

目标：形成完整闭环。

能力包括：

- 读取模型上下文。
- 生成细节设计计划。
- 做风格一致性检查。
- 做制作风险检查。
- 输出 Blender 辅助数据。
- 记录设计历史并支持迭代。

## 8. 成功指标

### 8.1 功能指标

- 常见机甲部件识别准确。
- 常见中文建模意图能解析为结构化计划。
- 输出计划包含明确操作步骤和风险提示。
- 能处理对称、成组和父子部件目标。

### 8.2 用户体验指标

- 用户不需要知道内部部件英文名。
- 输出内容能直接指导 Blender/ZBrush/Fusion 中的下一步操作。
- 对不合理请求能给出清晰原因和替代建议。
- 多轮修改后风格不明显漂移。

### 8.3 产品差异化指标

- 相比通用 3D AIGC，更懂机甲部件语义。
- 相比普通聊天助手，输出更结构化、可验证、可接工具。
- 相比纯建模脚本，更能处理自然语言和设计意图。

## 9. 风险与应对

### 9.1 风险：过早进入复杂 mesh 操作

应对：先做计划、审查、标记和辅助输出，再逐步接真实几何操作。

### 9.2 风险：功能范围过大

应对：优先围绕机甲细节设计、分件建议和可制作性检查，不做通用 3D 资产生成。

### 9.3 风险：版权和 IP 问题

应对：产品定位为原创机甲设计助手，不提供复刻具体商业机体的默认流程。

### 9.4 风险：LLM 输出不稳定

应对：用严格 schema、枚举值、校验器和 reviewer 限制输出，必要时自动修复或降级。

## 10. 下一步建议

具体任务：

1. 重新设计 `OperationPlan` 数据结构。
2. 扩展设计 taxonomy。
3. 更新 intent prompt。
4. 更新 mock parser 和 planner 校验。
5. 增加 5 到 10 条典型用例。
6. 增加设计师可读摘要和基础风险提示。
7. 跑通从用户输入到操作蓝图输出的闭环。

V0.2 完成后，这个项目就会从“基础意图 Agent”进入“自然语言到机甲设计操作蓝图”的阶段。
